# Secrets Manager 기반 CI/CD 시크릿 관리 설계

**작성일**: 2026-04-28
**브랜치**: main
**관련 이슈/PR**: TBD

## 1. 배경 및 목적

### 현재 상태
- `application-{env}.yml` 전체를 GitHub Repository Secret(`APPLICATION_DEV_YML`)에 통째로 저장하고, CI 단계에서 파일로 떨어뜨려 빌드에 포함하는 구조.
- 일부 값(DB)만 AWS Secrets Manager에서 런타임에 import (`spring.config.import: aws-secretsmanager:{project}/prod/db`).
- JWT secret, OAuth client id, Firebase 설정, API key 등은 yml 파일 본문에 평문 또는 placeholder로 존재.
- 운영용 AWS access-key/secret-key는 GitHub Secret에 저장되어 있고, CI에서 Docker build-args로 이미지에 baked되며, 런타임에도 yml의 `aws.credentials.access-key/secret-key`로 주입됨.

### 목적
- GitHub Secrets에 두는 시크릿을 **AWS 진입권(`AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`)** 만으로 최소화한다.
- 그 외 모든 시크릿을 **AWS Secrets Manager**에서 런타임에 로딩한다.
- 런타임에 정적 AWS 키가 코드/이미지/환경변수에 남지 않도록 ECS Task IAM Role 기반으로 전환한다.

## 2. 핵심 원칙

1. **GitHub Secrets는 AWS 진입권만 보유** — `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY` 두 개. 다른 시크릿은 일절 두지 않는다.
2. **런타임 정적 키 0개** — ECS Task Role의 임시 자격증명으로 SM 접근. yml의 `aws.credentials.access-key/secret-key` 블록 완전 제거. AWS SDK 클라이언트는 모두 `DefaultCredentialsProvider` 사용.
3. **yml은 git에 평범한 파일로 커밋** — `${...}` placeholder만 있고 평문 시크릿 0개. PR diff/리뷰 가능.

## 3. 시크릿 흐름 (아키텍처)

```
[GitHub Secrets]                          [AWS Secrets Manager]
  AWS_ACCESS_KEY_ID                          {project}/{env}/db        (RDS attached)
  AWS_SECRET_ACCESS_KEY                      {project}/{env}/secrets   (placeholder + auto api-key)
       │                                     {project}/{env}/jwt       (auto-generated)
       │                                     {project}/{env}/firebase  (Service Account JSON)
       │ (CI/CD 파이프라인용)                       │
       ▼                                            │
  [GitHub Actions]                                  │
  - aws-actions/configure-aws-credentials           │
  - ECR push, ECS deploy                            │
                                                    │
  [Docker Image] ── push ──> [ECR]                  │
                                                    │
  [ECS Task] ◀──── pull image                       │
       │                                            │
       │ TaskRole (secretsmanager:GetSecretValue)   │
       └────────── spring.config.import ───────────▶┘
                   (Task Role 임시 자격증명 자동 사용)
```

## 4. AWS Secrets Manager 시크릿 스키마

총 4개 시크릿. 도메인/회전 단위/자동 생성 가능 여부에 따라 분리.

### 4.1. `{project}/{env}/db`
- **관리 주체**: `cloudformation/rds.yml` (기존 `RDSSecretDev` 리소스, 변경 없음)
- **회전**: RDS와 SecretTargetAttachment로 연결되어 자동 회전 호환
- **키**:
  ```json
  {
    "host": "...",
    "port": "5432",
    "dbname": "...",
    "username": "...",
    "password": "..."
  }
  ```

### 4.2. `{project}/{env}/secrets`
- **관리 주체**: `cloudformation/ecs.yml` (`AppSecrets` 리소스, 신규)
- **회전**: 수동
- **키**:
  ```json
  {
    "google-client-id":  "REPLACE_ME",
    "apple-client-id":   "REPLACE_ME",
    "api-key":           "<auto-generated 50자>"
  }
  ```
- `api-key`는 CloudFormation `GenerateStringKey`로 자동 생성. EventBridge `RuleConnection`이 즉시 참조 가능해야 하므로 자동 생성 유지.
- `google-client-id`/`apple-client-id`는 외부 콘솔에서 받아오는 값이므로 placeholder.

### 4.3. `{project}/{env}/jwt`
- **관리 주체**: `cloudformation/ecs.yml` (`JwtSecret` 리소스, 신규)
- **회전**: 수동 (회전 시 모든 발급된 JWT 무효화됨에 유의)
- **키**:
  ```json
  {
    "jwt-secret": "<auto-generated 64자>"
  }
  ```
- 단일 키 자동 생성. 별도 시크릿으로 분리한 이유는 CloudFormation `GenerateSecretString`이 시크릿당 자동 생성 키 1개만 지원하기 때문.

### 4.4. `{project}/{env}/firebase`
- **관리 주체**: `cloudformation/ecs.yml` (`FirebaseSecret` 리소스, 신규)
- **회전**: 수동
- **포맷**: Service Account JSON 통째 저장
  ```json
  {
    "type": "service_account",
    "project_id": "REPLACE_ME",
    "private_key_id": "REPLACE_ME",
    "private_key": "REPLACE_ME",
    "client_email": "REPLACE_ME",
    "client_id": "REPLACE_ME"
  }
  ```
- **로딩 방식**: `spring.config.import`가 아니라 `SecretsManagerConfig.getSecret()` + `ObjectMapper` 파싱. 중첩/특수문자 포함 JSON이라 평탄 매핑에 부적합. (Firebase 코드 도입 시점에 구현)

## 5. yml 파일 재구성

### 5.1. 파일 구조
```
src/main/resources/
  application.yml          # 공통 + profiles 분기 (변경 없음)
  application-local.yml    # 신규 (dev SM 사용)
  application-dev.yml      # 신규
  application-prod.yml     # 신규
```
기존 `application-{env}.yml`은 삭제. (이름 자체가 Spring 인식 불가, 템플릿 자리표시였음)

### 5.2. `application-dev.yml` 핵심 부분
```yaml
app:
  env: dev

spring:
  cloud:
    aws:
      region:
        static: ap-northeast-2
      # credentials 블록 제거 — Task Role 자동 사용
  config:
    import:
      - aws-secretsmanager:{projectName}/dev/db
      - aws-secretsmanager:{projectName}/dev/secrets
      - aws-secretsmanager:{projectName}/dev/jwt
  datasource:
    driver-class-name: org.postgresql.Driver
    url: jdbc:postgresql://${host}:${port}/${dbname}
    username: ${username}
    password: ${password}

jwt:
  header: Authorization
  secret: ${jwt-secret}
  access-token-validity-in-milliseconds: 604800000
  refresh-token-validity-in-milliseconds: 7776000000
  algorithm: HS256

google:
  oauth:
    client-id: ${google-client-id}

apple:
  oauth:
    client-id: ${apple-client-id}

aws:
  # credentials 블록 통째로 제거
  region:
    static: ap-northeast-2
  s3:
    bucket: {projectName}-dev-bucket
    bucket-full-path: "https://${aws.s3.bucket}.s3.amazonaws.com/_media"
    bucket-path: "https://${aws.s3.bucket}.s3.ap-northeast-2.amazonaws.com/"

api-key:
  secret-key: ${api-key}
```

`firebase` 블록은 yml에서 제거 (코드에서 직접 SM fetch + 파싱).

### 5.3. `application-prod.yml`
`application-dev.yml`과 거의 동일하되:
- `app.env: prod`
- `spring.config.import` 경로의 `dev` → `prod`
- `aws.s3.bucket: {projectName}-prod-bucket`
- `cors.allowed-origins`에 prod 도메인

### 5.4. `application-local.yml`
- 로컬 개발자도 dev SM 사용 (Q4 결정사항).
- 자격증명: `~/.aws/credentials`의 프로필 사용.
- 다중 프로필 충돌 방지를 위해 **`AWS_PROFILE` 환경변수로 프로필 통제**.
  - `DefaultCredentialsProvider` 체인이 `AWS_PROFILE`을 자동 인식.
  - 각 개발자는 IDE Run Configuration 또는 셸에서 `AWS_PROFILE=<자기 프로필명>` 지정.
  - yml에는 프로필명 박지 않음 (개발자별로 다름).
- `spring.config.import`는 `dev` 시크릿 가리킴 (`aws-secretsmanager:{projectName}/dev/db|secrets|jwt`).

#### 로컬 사용 시 알려진 trade-off (수용)
1. 로컬에서 dev RDS에 직접 연결됨 (팀 작업 충돌 가능).
2. dev SM 장애/오프라인 시 로컬 부팅 불가.
3. 모든 개발자에게 dev 시크릿 노출 (IAM User 권한 운영 필요).

## 6. 애플리케이션 코드 변경

### 6.1. `S3Config.java` — Task Role 모드로 전환
```java
@Configuration
public class S3Config {

    @Value("${aws.region.static}")
    private String region;

    @Bean
    public S3Client s3Client() {
        return S3Client.builder()
            .region(Region.of(region))
            .credentialsProvider(DefaultCredentialsProvider.create())
            .build();
    }

    @Bean
    public S3Presigner s3Presigner() {
        return S3Presigner.builder()
            .region(Region.of(region))
            .credentialsProvider(DefaultCredentialsProvider.create())
            .build();
    }
}
```
- 기존 `@Value("${aws.credentials.access-key|secret-key}")` 필드 2개 제거.
- `@PostConstruct` `getAwsBasicCredentials()` 제거.
- `awsBasicCredentials` 필드 제거.
- `StaticCredentialsProvider` → `DefaultCredentialsProvider`.

### 6.2. `SesConfig.java` — 동일하게 전환
```java
@Configuration
public class SesConfig {

    @Value("${aws.region.static}")
    private String region;

    @Bean
    public SesClient sesClient() {
        return SesClient.builder()
            .region(Region.of(region))
            .credentialsProvider(DefaultCredentialsProvider.create())
            .build();
    }
}
```

### 6.3. `SecretsManagerConfig.java` — 변경 없음
- 이미 명시적 credentials provider 미지정 → `DefaultCredentialsProvider` 자동 사용.
- Firebase JSON 파싱 코드는 이 빈을 활용하여 추후 추가 (이번 작업 범위 외).

### 6.4. dead code 정리 (선택, 같은 PR에서 같이 정리 권장)
- `api-key.secret-name`, `api-key.secret-key` 코드 사용처 0개 확인됨 → yml에서 한 줄(`${api-key}`)로 단순화 시 영향 없음.
- `SecretsManagerConfig.getSecret()` 호출처 0개 — 향후 Firebase에서 사용 예정이므로 유지.

## 7. CloudFormation 변경

### 7.1. 변경 파일: `.github/cloudformation/ecs.yml`

기존 `Secret` 리소스 삭제 후 다음 3개로 재구성.

```yaml
AppSecrets:
  Type: AWS::SecretsManager::Secret
  Properties:
    Name: !Sub '${ProjectName}/${ENV}/secrets'
    GenerateSecretString:
      SecretStringTemplate: '{"google-client-id":"REPLACE_ME","apple-client-id":"REPLACE_ME"}'
      GenerateStringKey: 'api-key'
      PasswordLength: 50
      ExcludeCharacters: '"@/\&;='

JwtSecret:
  Type: AWS::SecretsManager::Secret
  Properties:
    Name: !Sub '${ProjectName}/${ENV}/jwt'
    GenerateSecretString:
      SecretStringTemplate: '{}'
      GenerateStringKey: 'jwt-secret'
      PasswordLength: 64
      ExcludeCharacters: '"@/\&;='

FirebaseSecret:
  Type: AWS::SecretsManager::Secret
  Properties:
    Name: !Sub '${ProjectName}/${ENV}/firebase'
    SecretString: '{"type":"service_account","project_id":"REPLACE_ME","private_key_id":"REPLACE_ME","private_key":"REPLACE_ME","client_email":"REPLACE_ME","client_id":"REPLACE_ME"}'
```

`RuleConnection` 참조 변경:
```yaml
# Before
ApiKeyValue: !Join [ '', [ '{{resolve:secretsmanager:', !Ref Secret, ':SecretString:key}}' ] ]

# After
ApiKeyValue: !Join [ '', [ '{{resolve:secretsmanager:', !Ref AppSecrets, ':SecretString:api-key}}' ] ]
```

### 7.2. TaskRole — 변경 없음
- `cloudformation/ecs.yml`의 `TaskRole`이 이미 `secretsmanager:GetSecretValue` 등 권한을 보유 (`Resource: '*'`).
- `Resource` 좁히기는 본 작업 범위에서 제외 (사용자 결정).

### 7.3. `cloudformation/rds.yml` — 변경 없음
- `{project}/{env}/db` 시크릿이 RDS와 attached 상태로 잘 굴러감.

## 8. CI/CD 워크플로 변경

### 8.1. `.github/workflows/ci_cd.yml`

#### 변경 1. "Copy Spring .yml" 단계 삭제
yml이 git에 커밋되므로 GitHub Secret에서 가져올 필요 없음.

#### 변경 2. Docker build-args 정리
`Dockerfile`이 어떤 build-arg도 사용하지 않음 (확인 완료). 4개 모두 dead code 제거:
```yaml
# After
- name: Build & Push Backend Image
  uses: docker/build-push-action@v4
  with:
    context: .
    push: true
    tags: >-
      ${{ steps.login-ecr.outputs.registry }}/${{ env.PROJECT_NAME }}-backend:${{ env.PROJECT_ENV }},
      ${{ steps.login-ecr.outputs.registry }}/${{ env.PROJECT_NAME }}-backend:${{ github.sha }}
    cache-from: type=gha
    cache-to: type=gha,mode=max
```
부수 효과: AWS 키가 이미지 layer history에 남던 보안 문제 자연 해소.

#### 변경 3. AWS credentials 단계 — 그대로 유지
ECR 로그인, ECS deploy에 필요. GitHub Secret의 AWS 키는 CI/CD 파이프라인 전용으로 유지.

#### 변경 4. ECS task definition / Service deploy — 그대로 유지
`SPRING_PROFILES_ACTIVE`만 환경변수로 주입하는 기존 구조 유지. 시크릿은 런타임에 Spring이 SM에서 직접 로딩.

### 8.2. `.github/workflows/initialize_project.yml`
신규 yml 3개의 `{projectName}` placeholder 치환 단계 추가:

```yaml
- name: Change
  run: |
    sed -i "s|spring_init|${PROJECT_NAME}|g" settings.gradle
    sed -i "s|spring_init|${PROJECT_NAME}|g" build.gradle
    sed -i "s|#{PROJECT_NAME}|${PROJECT_NAME}|g" README.md
    # 신규 추가
    sed -i "s|{projectName}|${PROJECT_NAME}|g" src/main/resources/application-local.yml
    sed -i "s|{projectName}|${PROJECT_NAME}|g" src/main/resources/application-dev.yml
    sed -i "s|{projectName}|${PROJECT_NAME}|g" src/main/resources/application-prod.yml
```

치환 대상 라인:
- `aws-secretsmanager:{projectName}/{env}/db|secrets|jwt`
- `aws.s3.bucket: {projectName}-{env}-bucket`

## 9. 마이그레이션 절차

각 환경(dev, prod)마다 동일 절차 수행. dev 먼저 검증 후 prod 진행.

### Phase 1. 코드 PR 작성 (머지 X)
다음 변경을 한 PR에 묶음:
- `application-{env}.yml` 삭제
- `application-local.yml`, `application-dev.yml`, `application-prod.yml` 신규
- `S3Config.java`, `SesConfig.java` Task Role 모드 전환
- `cloudformation/ecs.yml` 시크릿 리소스 재구성 + `RuleConnection` 참조 변경
- `.github/workflows/ci_cd.yml` "Copy Spring .yml" 삭제 + build-args 정리
- `.github/workflows/initialize_project.yml` yml 치환 단계 추가

### Phase 2. CloudFormation 인프라 업데이트 (대상 환경)
`build_ecs.yml` 워크플로 수동 트리거 또는 AWS CLI:
- 새 시크릿 3개 생성 (`AppSecrets`, `JwtSecret`, `FirebaseSecret`)
- `RuleConnection`이 새 시크릿의 `api-key`로 참조 갱신
- 기존 `${project}/{env}/spring` 삭제 (30일 복구 윈도우 보유)

이 시점에 ECS 컨테이너는 옛 이미지로 동작 중. 옛 이미지는 `${project}/{env}/spring`을 코드에서 호출하지 않으므로 (dead code 확인됨) 영향 없음.

### Phase 3. SM 콘솔에서 실제 값 입력
- `{project}/{env}/secrets`: `google-client-id`, `apple-client-id` 입력. `api-key`는 자동 생성 그대로.
- `{project}/{env}/jwt`: 자동 생성된 64자 값 확인만.
- `{project}/{env}/firebase`: Firebase 도입 시점까지는 placeholder OK.
- `{project}/{env}/db`: rds.yml이 관리, 손대지 않음.

### Phase 4. PR 머지 → CI/CD 자동 배포
- develop(또는 main) 브랜치 머지 → CI 동작 → ECR push → ECS deploy.
- 새 컨테이너가 부팅하며 `spring.config.import`로 SM 값 로딩.

### Phase 5. 검증

| 항목 | 확인 방법 |
|---|---|
| CloudFormation update | stack status `UPDATE_COMPLETE` |
| SM 시크릿 3개 신규 생성 | 콘솔에서 `secrets`, `jwt`, `firebase` 존재 확인 |
| SM placeholder 잔존 여부 | 모든 값이 `REPLACE_ME` 아님 (firebase는 예외) |
| ECS task 신규 revision 배포 | task definition revision 증가 |
| 컨테이너 부팅 | CloudWatch 로그에 `Started SpringInitApplication` |
| `/health` | 200 OK |
| DB 연결 | 임의 GET API 정상 응답 |
| JWT 로그인 | 로그인 API 토큰 발급 |
| OAuth 로그인 | Google/Apple 소셜 로그인 1회 |
| S3 업로드 | presigned URL 발급 + 업로드 |
| SES 발송 | 이메일 인증 1회 |
| EventBridge 트리거 | (해당 시) ALB 도달 확인 |
| CloudWatch 에러 로그 | 부팅 후 5분간 ERROR 없음 |

### Phase 6. prod 동일 절차 + 후속 정리
- Phase 2~5를 `ENV=prod`로 반복.
- GitHub Repository Secret `APPLICATION_DEV_YML`(과 prod용이 있다면 그것도) 콘솔에서 삭제.
- 옛 SM 시크릿 30일 복구 윈도우 만료 후 영구 삭제 자동 처리.

## 10. Rollback 계획

| 시나리오 | 대응 |
|---|---|
| 코드 결함 | PR `git revert` → CI 재실행 → 옛 이미지로 복귀 (옛 이미지는 SM 의존성 없는 yml 보유) |
| 인프라 결함 | CloudFormation stack 이전 버전으로 롤백 (콘솔) |
| SM 값 오입력 | 콘솔에서 값 수정 → ECS service force-new-deployment |
| 옛 시크릿 복구 필요 | 30일 윈도우 내 콘솔에서 복원 |

## 12. 결정 기록 (Q&A 요약)

| # | 질문 | 결정 | 근거 |
|---|---|---|---|
| 1 | yml 통째 GitHub Secret vs git 커밋 | git 커밋 | PR 리뷰 가능, 시크릿 분리 |
| 2 | ECS 런타임 인증 방식 | Task Role | 정적 키 노출 0, best practice |
| 3 | SM 시크릿 그룹핑 | 도메인별 분리 (db / secrets / jwt / firebase) | 회전 단위 명확, RDS 회전 호환, jwt 자동 생성 |
| 4 | 로컬 개발 SM 사용 여부 | 사용 (dev SM) | 환경 일관성. trade-off 수용 |
| 5 | SM 값 채우는 방식 | CloudFormation placeholder + 운영자 수동 입력 + 일부 자동 생성 | IaC에 평문 시크릿 안 들어감 |
| 6 | TaskRole `Resource` 좁히기 | 미적용 | 본 작업 범위 외 |
| 7 | jwt-secret 자동 생성 | 적용 (별도 시크릿으로 분리) | CFN 자동 생성 키 1개/시크릿 제약 회피 |
| 8 | Firebase 시크릿 분리 | 분리 (`{project}/{env}/firebase`) | Service Account JSON 구조 (중첩) → 자동 매핑 부적합 |
| 9 | 로컬 다중 AWS 프로필 통제 | `AWS_PROFILE` 환경변수 | AWS SDK 표준, 코드/yml 수정 불필요 |
